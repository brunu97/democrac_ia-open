Como Funciona

Como é que funciona o Democrac_IA de forma direta

O Democrac_IA é como um bibliotecário inteligente que sabe tudo sobre os debates da Assembleia da República desde Dezembro de 2006 até Janeiro de 2026.

Graças ao tabalho árduo das pessoas que fazem de transcrição e organização dos documentos parlamentares, que podes encontrar aqui parlamento.pt, o Democrac_IA tem acesso a uma vasta coleção de textos. Que foram todos transferidos para serem usados nesta aplicação.

O Democrac_IA NÃO faz pedidos diretos ao servidor do parlamento para obter informações. Em vez disso, usa uma cópia local no proprio servidor do Democrac_IA dos documentos que foram processados e organizados antecipadamente.

Em teoria esta aplicação poderia usar documentos desde 1821 mas devido a limatações tecnicas é focada em dados mais recentes e isto ajuda-te a encontrar respostas sobre o que os deputados dizem, votam e discutem.

Imagina que tens milhares de livros (documentos de sessões parlamentares) e queres saber algo específico. O Democrac_IA lê rapidamente todos os livros e conta-te exatamente o que encontraste.

Se tens uma dúvida ou estás à procura de algum contacto vai aqui

1. Fazes uma pergunta

Como "Quem votou contra a aquela coisa que aconteceu naquela altura?" ou "O que disse o deputado X sobre o assunto daquela coisa?".

2. A app procura nos documentos

O Democrac_IA tem uma "biblioteca" com milhares de páginas de debates parlamentares. Ele usa magia matemática para encontrar as partes mais relacionadas com a tua pergunta.

3. Entra a IA para perceber a resposta

Um modelo de linguagem avançado entra no cenário e lê essas partes e cria uma resposta clara sobre o tema.

4. Vês a resposta e as fontes

Para dar confiança na resposta, o Democrac_IA indica exatamente de onde veio a informação. É mostrado as fontes com links para os documentos originais. Podes clicar e ver exatamente de onde veio a informação.

Como é que o Democrac_IA realmente funciona

Queres mesmo saber o que realmente se passa aqui?

Primeiro de tudo, tudo isto é codigo aberto! o backend e o frontend, vai aqui, podes fazer o que quiseres, podes copiar, expandir, tanto faz GitHub

O código não está perfeito, e certamente há pontos onde é possivel melhorar, se tens alguma sugestão, envia para aqui

Primeiro vamos á maneria como o site funciona e de seguida á forma como os dados foram indexados

A Pesquisa no site de forma resumida num diagrama funciona assim, é essencialmente um RAG (Retrieval-Augmented Generation)


Toda a backend é feita em Python usando Flask, Python é ideial tudo o que é IA e ML, há exelentes modulos para se usar.

Um dos módulos utilizados é o SentenceTransformer, que também é responsável pela geração dos embeddings. Escolhi o BGE-M3 pois com base nos meus testes, apresenta um desempenho ok para Português de Portugal mas uma boa alternativa seria o E5-Multilingual-Large, no entanto existem modelos maiores mas como isto será executada via CPU num servidor comum. Por questões de performance, é necessário um modelo de dimensões médias e tanto o E5-Multilingual-Large como o BGE-M3 encaixam-se perfeitamente neste requisito e ambos são open source!

Depois de transforar os textos em vetores, estes beep boops podem ir para o FAISS fazer a pesquisa.


O FAISS

Facebook AI Similarity Search ou simplesmente FAISS é uma biblioteca de pesquisa vetorial desenvolvida pela Meta pela equipa do Facebook IA (dai o nome). É usada para encontrar os vetores mais semelhantes a um vetor de consulta em uma grande base de dados vetoriais.

Não é exatamente uma Inteligência Artificial por si, mas sim uma biblioteca extremamente eficiente a encontrar dados

O Resultado do embedding do passo anterior é utilizado aqui para o FAISS procurar dados relevantes

FAISS é Open Source e existem outras altenativas, tal como o ChromaDB, no entanto FAISS é incrivelmente rapido e leve e foi a razão de o ter escolhido

Todos os textos estão guardados numa base dados em SQLite, textos estes que são Chunks dos documentos originais que foram indexados


A Mágia acontece - LLM com RAG

Depois de se encontrar toda a informação relevante, todo os dados incluindo um conjunto de instruções destinadas ao system prompt da LLM é enviado para esta fase do processo

O sucesso da resposta depende diretamente da qualidade das fontes utilizadas e do system prompt que orienta o comportamento do modelo. O ficheiro de prompts descreve de forma detalhada como a IA deve analisar as fontes e estruturar as suas respostas

Inicialmente, isto foi desenvolvida com o Ollama a funcionar como servidor local. No entanto, para um ambiente de produção, o Ollama não é a opção mais adequada. A alternativa é o vLLM que é ideal para aplicações web com múltiplos utilizadores, mas o custo é bastante elevado para manter um servidor desses assim

Por esse motivo esta aplicação recorre ao groq (sem qualquer relação com o Grok do X/Twitter). groqCloud É um serviço rápido e económico, o que o torna uma excelente escolha para estes cenários

Durante a fase de desenvolvimento, o modelo escolhido foi o Qwen 2.5 7B, desenvolvido pela Alibaba que também é open source, tal como todas as restantes tecnologias utilizadas até aqui.

Mas porquê um modelo tão pequeno? perguntas tu! Bem... hospedar modelos maiores é caro 😅, MAS o Qwen 2.5 7B é excelente na comunicação em português de Portugal. Uma alternativa viável seria o LLaMA 2 7B, da Meta, mas nos testes realizados o Qwen demonstrou ser significativamente melhor a seguir instruções, qualquer das formas a ideia de usar um servidor dedicado com a vLLM acabou por ser abandonada devido aos custos elevados de manter um servidor dedicado a executar estas LLMs, mesmo sendo modelos bastante eficientes e leves.

Atualmente, a aplicação realiza a sua inferência diretamente no groqCloud, utilizando o LLaMA 3.1 8B (Qwen 2.5 7B não está disponível no groq) que é barato e rapido. O serviço também suporta o Qwen 3 32B, que poderá vir a ser utilizado no futuro caso seja necessário, mas também existem outros modelos no groq, no entanto, para o contexto desta aplicação, apenas estes dois modelos foram testados.

Todo o frontend é feito em Angular, não haveria nada de mal em utilizar o Flask diretamente com templates para fazer o site, mas Angular é mais facil de gerir

A Extração de dados

Talvez a parte mais difícil seja a extração dos PDFs dos debates do Parlamento. Inicialmente, a ideia era bem para além de só dos debates, mas como todo o conteúdo que o site tem, desde 1821 (aproveito já para agradecer o trabalho incrível que o pessoal do parlamento fez a transcrever tudo isto e ter tudo digitalizado), dificuldades técnicas surgiram rapidamente. Não só a resolução de vários ficheiros não é a ideal para um computador ler, como o esforço do hardware para processar isto seria enorme, então decidi focar só nos debates a partir de 2006.

Todos os PDFs foram extraídos com o devido cuidado do site do parlamento, com um script de crawl que demorou imenso tempo a percorrer o site e encontrar os ficheiros de debates e fazer download um a um dos PDFs. No total, foram cerca de 1900 PDFs.

Depois disso estar feito, entra a parte de transformar isto tudo em embeddings usando pytorch e o modelo BGE-M3, coisa que felizmente posso acelerar com ROCm. Sendo o meu GPU da AMD, uma RX 7800XT, posso usar o GPU para fazer isto mais rápido. Originalmente o processo demorava mais de 10 horas seguidas 😫! mas depois de varias experiências e optimizações de momento demora cerca de uma hora e meia! CUDA seria sempre a opção ideal mas requer GPU NVIDIA.

A estrura de cada PDF é semelhante ao longo dos anos dos documentos, no entanto há algumas alterações, como por exemplo a lista de presença existe nos mais antigos mas não nos novos, mas a variação entre pdfs não é suficiente para causar ruidos significativos na pesquisa

Tudo é armazenado na base de dados e indexado no FAISS.
